Guardianity — Hackathon demo

Give any language model a conscience, without retraining it. Your model stays as it is; our heart reads each turn and decides — allow, refuse with care, or shape the answer. The model is yours; the heart is ours, hosted, and never exposed in this folder.

What this is

This folder is a small, runnable client. It lets you chat with a model governed by the Guardianity heart in real time — and flip the heart off to see the raw model for comparison. It does not contain the heart: the conscience runs on our live hosted service and is reached over its API. The full system (the Umbra language and the rest) is not published — this is a faithful way to test the heart on your model.

How it works

   YOUR MACHINE                                  OUR LIVE SERVICE        
 ┌───────────────────────────┐                 ┌────────────────────────┐
 │  this folder              │  POST /kagune/turn (Bearer key)          │
 │  ├─ your model            │ ───────────────▶ │  the Guardianity heart │
 │  ├─ guardianity_client.py │ ◀─────────────── │  (Scale · gate · warm  │
 │  └─ console + docs        │   decision +     │   refusal · steering)  │
 │                           │   modulation     │                        │
 └───────────────────────────┘                 └────────────────────────┘
        the model is yours                          the conscience is ours

Each turn: you type → the demo asks the hosted heart how to handle it → if allowed, your model answers (with the heart's voice modulation); if refused with care, your model writes the refusal in its own voice from the heart's directive. The heart only ever sees the turn's text — your model and weights never leave your machine.

Two ways to connect a model

Pick whichever fits — set it in .env with VESSEL_MODE:

ModeVESSEL_MODEWhat runs where
Direct / locallocal Your model runs in this folder (downloaded from Hugging Face). Simplest — one command and you're chatting. Great on a GPU like an A100.
Remote vesselremote Your model runs on your own GPU elsewhere, behind an HTTP endpoint; you give its URL (VESSEL_URL). The demo calls it to generate.

In both modes the heart is the same hosted service — only where your model runs changes.

Getting your key

  1. Go to the Renji / Guardianity website and create an account (during the event, verification is open — just sign up).
  2. Copy your API key and the hosted-heart URL.
  3. Choose your connection mode (direct or remote) — see above.

There is no shared key: each tester uses their own account and key.

Setup & run

cp .env.example .env        # then fill it in (below)
./run.sh                    # installs deps, launches; your model downloads on first boot
# open http://localhost:8011

In .env:

KeyMeaning
RENJI_KEYyour API key (from the site)
RENJI_URLthe live hosted-heart URL (from the site)
VESSEL_MODElocal or remote
HACK_MODEL_ID(local) your model — default is our abliterated model; swap freely
HACK_DEVICE(local) cpu or cuda (use cuda on an A100)
VESSEL_URL(remote) your self-hosted vessel endpoint
HF_TOKENonly for gated models (e.g. Gemma)

The endpoint contract

POST {RENJI_URL}/kagune/turn · header Authorization: Bearer {RENJI_KEY}

You send:

{ "text": "the user's message" }

You get back:

{
  "decision": "allow" | "refusal" | "refusal-care" | "crisis" | "third-path",
  "refusal_text": "a ready fallback refusal",
  "refusal_directive": "meta-prompt: let YOUR model write the refusal in its voice",
  "voice_modulation": { "temperature_delta": -0.1, "top_p_delta": -0.05 },
  "vow_intent": "why", "trace_id": "id"
}

What to do with it: on allow/third-path, your model answers (apply the modulation deltas); on a refusal decision, let your model write the refusal from refusal_directive (or show refusal_text). That's the whole integration — see guardianity_client.py and app.py for the reference.

Remote-vessel contract (Mode B), for the model server you host: POST {VESSEL_URL} with { "prompt": "...", "max_tokens": 512, "temperature": 0.7 } → { "text": "..." } (OpenAI-style choices also accepted).

Heart ON / OFF

The console has a toggle. ON routes the turn through the hosted heart (governed). OFF runs your model raw, with no conscience — so you can see exactly what the heart changes. If the heart isn't reachable, turns are clearly marked offline · bypassed, so you always know whether the conscience was in the loop.

Privacy & limits

  • Your model and weights never leave your machine; the heart only sees the turn's text.
  • This is for testing during the event — an open trial.
  • Nothing here exposes the heart's internals. By design.
